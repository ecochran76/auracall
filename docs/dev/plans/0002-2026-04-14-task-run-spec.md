# Task / Run-Spec Plan

## Purpose

Define the first code-facing shape for the concrete assignment layer that sits
between a reusable `team` template and a durable execution `run`.

This plan exists to answer one practical question:

- what should AuraCall persist and transport as the concrete assignment for a
  team without overloading `team` itself or leaking execution-owned state into
  a reusable template

It should be read together with:

- [0006-2026-04-14-team-config-boundary.md](/home/ecochran76/workspace.local/oracle/docs/dev/plans/0006-2026-04-14-team-config-boundary.md)
- [0004-2026-04-14-team-service-execution.md](/home/ecochran76/workspace.local/oracle/docs/dev/plans/0004-2026-04-14-team-service-execution.md)

## Canonical split

AuraCall should separate three layers:

1. `team`
   - reusable orchestration template
   - who collaborates and the default rules for collaboration
2. `task` / `run spec`
   - one concrete assignment for one team invocation
   - what work should be done this time
3. `run`
   - one durable execution attempt
   - what actually happened over time

Important rule:

- `task` / `run spec` is the executable assignment boundary
- public `team run` semantics should bind one task / run spec to one team
  template

## Why this layer is needed

Without a dedicated task layer, the repo risks collapsing three different
things together:

- reusable team policy
- one-off assignment detail
- execution history/state

That creates the wrong pressure:

- teams become too specific and hard to reuse
- runs have to reconstruct the original assignment from ad hoc fields
- public execution semantics drift toward the current MVP member-order
  projection instead of the real product model

## Design goals

The first task / run-spec shape should optimize for:

- explicit ownership
- serializability
- compatibility with CLI, API, MCP, and service execution
- stable handoff into the team planner
- support for unattended multi-turn automation

It should not optimize first for:

- provider-specific request shaping
- hidden browser state
- maximal schema richness
- speculative planner intelligence

## What belongs on `team`

Team-owned defaults should include:

- member roles
- team-level instructions
- delegation policy
- response-shape contracts
- handoff policy
- default automation policy
- default stop/escalation policy
- default local host-action policy

These are reusable collaboration defaults.

## What belongs on `task` / `run spec`

Task-owned fields should include:

- assignment identity
- selected team id
- input bundle references
- concrete objective
- success criteria
- task-specific context and constraints
- requested outputs
- task-level overrides of team defaults when allowed
- requester / trigger context

These are one-off assignment details.

## What belongs on `run`

Run-owned fields should include:

- durable run id
- lifecycle timestamps
- step execution state
- handoffs actually produced
- artifacts actually produced
- local actions actually requested/executed
- stop/failure/intervention history

These are execution facts, not assignment intent.

## Recommended first shape

The first bounded shape should be one top-level `taskRunSpec` object with six
explicit sections:

- identity
- assignment
- inputs
- execution
- output
- provenance

Required top-level fields for v1:

- `id`
- `version`
- `teamId`
- `assignment`
- `inputs`
- `execution`
- `output`
- `provenance`

## First concrete v1 contract

Recommended bounded TypeScript-style shape:

```ts
type TaskRunSpec = {
  id: string;
  version: 1;
  teamId: string;
  assignment: {
    title: string;
    objective: string;
    successCriteria: string[];
    context?: Record<string, unknown> | null;
    constraints?: {
      maxTurns?: number | null;
      maxRuntimeMinutes?: number | null;
      allowedServices?: string[] | null;
      blockedServices?: string[] | null;
    } | null;
  };
  inputs: {
    prompt?: string | null;
    structuredContext?: Record<string, unknown> | null;
    inputArtifacts: TaskRunInputArtifact[];
  };
  execution: {
    mode: 'sequential';
    failPolicy: 'fail-fast';
    humanInteractionPolicy: {
      defaultBehavior: 'pause' | 'fail' | 'continue';
      allowClarificationRequests: boolean;
      allowApprovalRequests: boolean;
    };
    localActionPolicy: {
      mode: 'forbidden' | 'allowed' | 'approval-required';
      allowedActionKinds?: string[] | null;
      allowedCommands?: string[] | null;
      allowedCwdRoots?: string[] | null;
      resultReportingMode: 'summary-only' | 'summary-and-payload';
    };
    overrides?: {
      runtimeProfileId?: string | null;
      browserProfileId?: string | null;
      promptAppend?: string | null;
    } | null;
  };
  output: {
    requestedOutputs: TaskRunRequestedOutput[];
  };
  provenance: {
    trigger: 'cli' | 'api' | 'mcp' | 'service' | 'scheduled' | 'internal';
    requestedBy?: {
      kind: 'user' | 'service' | 'schedule' | 'internal';
      id?: string | null;
      label?: string | null;
    } | null;
    createdAt: string;
    parentSessionId?: string | null;
  };
};
```

### Section rules

`identity`
- `id` and `version` are contract identity only
- do not put lifecycle state here

`assignment`
- owns the one-shot work definition
- `successCriteria` stays user- or caller-facing
- `context` is descriptive assignment context, not runtime state
- `constraints` are bounded run limits, not infrastructure topology

`inputs`
- `prompt` is the human-readable starting ask when one exists
- `structuredContext` is machine-readable assignment context
- `inputArtifacts` are durable references, not live browser/session handles

`execution`
- v1 hard-codes conservative semantics:
  - `mode = sequential`
  - `failPolicy = fail-fast`
- `humanInteractionPolicy` and `localActionPolicy` are assignment-level execution controls
- `overrides` are temporary per-run overrides only

`output`
- `requestedOutputs` describes the desired deliverable shape
- do not put produced artifacts or execution summaries here

`provenance`
- records who/what caused the task to exist
- remains serializable and transport-safe
- does not encode queue, worker, or lease ownership

## Explicit v1 non-goals

The first contract should not include:

- multi-team binding
- implicit parallelism
- worker placement or queue topology
- browser session handles or hidden DOM state
- step/handoff/shared-state execution artifacts
- provider-specific low-level request knobs unless a current runtime seam already requires them

## Boundary summary

Use this litmus test for every candidate field:

- reusable across many future invocations of the same team -> `team`
- specific to this one assignment -> `taskRunSpec`
- produced while work is executing -> `teamRun` / runtime

## Binding rule to team execution

The first execution binding should be explicit:

1. resolve one reusable `team`
2. accept one concrete `taskRunSpec`
3. derive one planned `teamRun`
4. project that planned `teamRun` into runtime execution records

Important rule:

- one `taskRunSpec` binds to one selected `team`
- one `teamRun` is one execution attempt of that bound pair
- retries, cancellation, and runner ownership belong to `teamRun` / runtime,
  not to `taskRunSpec`

This keeps assignment intent stable even when the durable run retries or fails.

## First implementation boundary

The first implementation should not try to model every future workflow.

Safe MVP constraints:

- one `taskRunSpec` maps to one team template
- one primary objective only
- `successCriteria` is a list of strings, not a custom expression language
- `context` is structured JSON only
- `requestedOutputs` is explicit and bounded
- no embedded binary payloads; use `inputArtifacts` refs instead
- `overrides` may narrow behavior, but should not replace team identity

That is enough to:

- bind a concrete assignment to a team
- support unattended multi-turn automation
- support later host-action and human-escalation policy checks
- preserve clean separation between assignment intent and execution state

Recommended semantics:

- `title`
  - short operator-readable assignment label
- `objective`
  - primary assignment statement
- `successCriteria`
  - deterministic completion criteria when available
- `requestedOutputs`
  - expected work products and response shape requirements
- `inputArtifacts`
  - durable refs to files, bundles, docs, prior artifacts, or other inputs
- `context`
  - structured assignment context that should be visible to the planner
- `constraints`
  - hard limits such as budget, services, time, provider restrictions, or
    allowed tools
- `overrides`
  - task-scoped overrides to team defaults where allowed
- `turnPolicy`
  - max turns, stop conditions, and loop behavior for this assignment
- `humanInteractionPolicy`
  - when AuraCall must stop for human review or approval
- `localActionPolicy`
  - whether host actions are allowed and which action classes are permitted

## Recommended helper shapes

### `requestedOutputs`

Use a list of explicit output contracts.

Each output contract should support:

- `kind`
- `label`
- `format`
- `required`
- `schemaHint`
- `destination`

Recommended first `kind` values:

- `final-response`
- `patch`
- `artifact-bundle`
- `review-note`
- `structured-report`

Recommended first `destination` values:

- `response-body`
- `artifact-store`
- `handoff`

This allows task-specific output requirements without baking them into the team
definition itself.

### `inputArtifacts`

Each input artifact should support:

- `id`
- `kind`
- `title`
- `path`
- `uri`
- `mediaType`
- `notes`
- `required`

Important rule:

- inputs should be durable refs, not implicit dependence on open browser tabs

### `turnPolicy`

The first task-level loop policy should support:

- `maxTurns`
- `stopOnStatus`
- `allowTeamInitiatedStop`
- `allowHumanEscalation`

This is the concrete per-assignment loop contract.

### `humanInteractionPolicy`

The first human-policy shape should support:

- `requiredOn`
  - statuses or conditions that require human input
- `allowClarificationRequests`
- `allowApprovalRequests`
- `defaultBehavior`
  - `pause`
  - `fail`
  - `continue`

### `localActionPolicy`

The first host-action shape should support:

- `mode`
  - `forbidden`
  - `allowed`
  - `approval-required`
- `complexityStage`
  - `bounded-command`
  - `repo-automation`
  - `extended`
- `allowedActionKinds`
- `allowedCommands`
  - optional per-task narrowing of concrete shell executables
- `allowedCwdRoots`
  - optional per-task narrowing of allowed working-directory roots
- `resultReportingMode`

Important rule:

- task policy should decide what is allowed for this assignment
- host/runtime policy should still own the hard execution ceiling
- run history should record what was actually requested and executed

## Recommended initial helper contracts

Recommended first helper shapes:

```ts
type TaskRunRequestedOutput = {
  kind: 'final-response' | 'patch' | 'artifact-bundle' | 'review-note' | 'structured-report';
  label: string;
  format: 'text' | 'markdown' | 'json' | 'diff' | 'bundle';
  required: boolean;
  schemaHint?: string | null;
  destination: 'response-body' | 'artifact-store' | 'handoff';
};

type TaskRunInputArtifact = {
  id: string;
  kind: 'file' | 'directory' | 'doc' | 'bundle' | 'prior-artifact' | 'url';
  title: string;
  path?: string | null;
  uri?: string | null;
  mediaType?: string | null;
  notes?: string[] | null;
  required: boolean;
};
```

These helper contracts are enough for the first team-binding slice without
committing the repo to a larger planner DSL.
