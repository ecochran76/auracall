# Windows work notes

Read this file whenever you're working from Windows and add new findings so the next agent can stay unblocked.

- Browser engine now allowed on Windows; expect more flakiness. If automation fails, rerun with `--engine api --wait` or point `--remote-chrome` to a running Chrome with remote debugging.
- Working integrated path (2026-03-25): from WSL, Aura-Call can now launch Windows Chrome directly with a managed Aura-Call profile rooted on Windows (for example `/mnt/c/Users/ecoch/AppData/Local/AuraCall/browser-profiles/<profile>/grok`), seed it from the Windows Chrome default profile, and drive Grok end to end without firewall or `portproxy` setup. Verified with a fresh managed profile plus `AURACALL_WSL_CHROME=windows`.
- Cleanup follow-up (2026-03-26): the remaining “Chrome stays open after a successful run” bug was a double-launch bug, not just a bad shutdown hook. The generic browser runner was prelaunching Chrome before handing Grok off to `runGrokBrowserMode(...)`, so the Grok path re-adopted the same Windows-managed profile as a reused instance and skipped the final kill. Grok now routes directly into its own local launch path, and browser-service preserves current-run ownership by DevTools port/PID when it has to re-adopt. Verified with a fresh Windows-managed profile (`windows-cleanup-proof-3`) that returned a real Grok answer and left no matching `chrome.exe` process behind.
- Path normalization follow-up (2026-03-26): browser-service now normalizes WSL/Linux paths, Windows drive paths (`C:\...`), and WSL UNC paths (`\\wsl.localhost\Ubuntu\...`) consistently across config resolution, managed-profile bootstrap, process matching, and Chrome launch. Users can keep supplying ordinary WSL paths from WSL; Aura-Call handles the `\\wsl.localhost\...` conversion for Windows Chrome automatically instead of requiring manual escaping.
- Browser-family discovery follow-up (2026-03-26): when WSL is targeting Windows Chromium, profile discovery now prefers the matching browser family from the configured executable hint (`chrome.exe` vs `brave.exe`) instead of always defaulting to the first Windows profile tree it finds. That makes Windows Brave/Chrome bootstrap much less configuration-sensitive.
- Windows Default-profile bootstrap follow-up (2026-03-26): managed-profile bootstrap now uses a Windows shared-read copy fallback for locked Chromium files, rewrites copied `Preferences` / `Local State` to mark clean exit, prunes copied session artifacts, and launches with `--hide-crash-restore-bubble`. That removed the earlier “restore pages / Chrome didn’t shut down correctly” modal from the fresh integrated proof path.
- Windows explicit-profile safety follow-up (2026-03-26): integrated Windows Chrome retries must not destructively reset an explicitly selected managed profile path. Aura-Call now preserves explicit `manualLoginProfileDir` selections across Windows DevTools port retries instead of deleting/reseeding them. This matters for real persistent profiles such as `/mnt/c/Users/ecoch/AppData/Local/AuraCall/browser-profiles/windows-default-import-4/grok`, where a user may have manually logged in.
- Grok auth verification follow-up (2026-03-26): Grok setup/login checks now require a positive signed-in identity, not just “no obvious login error.” A remote attach against the pinned Windows-managed Grok profile on `windows-loopback:45910` correctly blocked with `visible Sign in/Sign up controls still present`, so guest chat no longer counts as authenticated setup.
- Current local profile split (2026-03-26): `~/.auracall/config.json` now keeps separate Aura-Call profiles for the two browser systems while onboarding stabilizes. The default `wsl-chrome` profile points at the WSL-managed root under `~/.auracall/browser-profiles/...`; the Windows-specific profile (`--profile windows-chrome` / scratch `windows-chrome-test`) points at the Windows Aura-Call managed root under `/mnt/c/Users/ecoch/AppData/Local/AuraCall/browser-profiles/...`.
- Profile precedence follow-up (2026-03-26): selected Aura-Call profiles must override `browserDefaults`. A resolver bug meant `--profile windows-chrome-test` still inherited WSL Chrome defaults because `browserDefaults` had already filled `browser` and the profile overlay only filled missing fields. Fixed precedence to `CLI > selected profile > browserDefaults`, and extended profile-browser parsing so v2 profile blocks can use `chromeCookiePath` / `chromeProfile` directly.
- Manual-login endpoint follow-up (2026-03-26): the login/manual-launch path was printing and registering the requested debug port instead of the actual launched Windows port. On this machine that hid the real managed Windows browser behind a fake `windows-loopback:9222` while the live listener was actually `45920`. `packages/browser-service/src/login.ts` now uses `chrome.port` / `chrome.host` from the launched session, so `auracall login --target grok` reports the real endpoint.
- File-tab investigation follow-up (2026-03-26): the reproducible stray `file:///C:/Users/ecoch/AppData/Local/AuraCall/browser-profiles/windows-chrome-test/grok/` window did not come from the current product launcher. The bad window belonged to a second browser process with broken command line `--user-data-dir= C:\Users\ecoch\...\windows-chrome-test\grok --remote-debugging-port=45920 https://grok.com/` (note the space after `=`). The real DevTools listener on `127.0.0.1:45920` belonged to the good process `--user-data-dir=C:\Users\ecoch\...\windows-chrome-test\grok --remote-debugging-port=45920 about:blank`. After killing the broken sibling PID, the remaining live `json/list` tabs for the managed profile contained Grok pages plus `about:blank`, not the file URL.
- Manual-login tab reuse follow-up (2026-03-26): repeated `auracall --profile windows-chrome-test login --target grok` should not keep adding Grok tabs to the same live managed Windows profile. `packages/browser-service/src/manualLogin.ts` now reuses an existing matching login tab first, then reuses an existing `about:blank` page by navigating it, and only falls back to `CDP.New(...)` when there is nothing reusable. Verified live on the managed Windows profile at `127.0.0.1:45920`: Grok page count stayed `4 -> 4` across another login launch instead of growing again.
- DevTools rediscovery follow-up (2026-03-26): browser-service now has a profile-scoped Windows CDP recovery path. It checks the managed profile's recorded `DevToolsActivePort`, the advertised `--remote-debugging-port`, and any Windows-local listeners owned by Chrome processes for the same `user-data-dir`. If a different live port is found, Aura-Call rewrites `DevToolsActivePort` and adopts it instead of blindly trusting the requested port.
- Auto-port follow-up (2026-03-26): fresh WSL -> Windows Chrome launches should use `--remote-debugging-port=0` and adopt the real port from `DevToolsActivePort`. A Windows-side A/B test showed that literal PowerShell launches with correct quoting can expose CDP, but some nearby fixed ports fail on this machine (`45941` worked while `45942` timed out). Aura-Call now lets Windows choose the DevTools port and discovers it from the managed profile instead of pre-picking a fixed port from Linux.
- Real-profile follow-up (2026-03-26): after killing the stale fixed-port root for `windows-chrome-test/grok`, Aura-Call cleanly relaunched the real managed Windows profile with `--remote-debugging-port=0` and adopted `windows-loopback:49926` from `DevToolsActivePort`. The managed profile still reports the signed-in Chrome account (`Eric Cochran <ecochran76@gmail.com>`), so browser-profile persistence is working; the remaining problem was registry/liveness attribution.
- Liveness follow-up (2026-03-26): for managed Windows profiles, a responsive Windows-local `/json/version` endpoint is now treated as authoritative proof of life. This fixes the earlier false stale-entry report where the real `windows-chrome-test/grok` profile was live on `49926` but `auracall doctor --local-only` still showed `alive: false`.
- Primary Windows-from-WSL path (2026-03-26): the supported happy path is now managed Windows profile + `--remote-debugging-port=0` + `DevToolsActivePort` discovery + Aura-Call’s built-in `windows-loopback` relay. Firewall rules and `portproxy` are not required for this product path.
- Bot-detection follow-up (2026-03-26): the live Windows-managed Chrome profile uses a normal stock UA string (`Chrome/146` on `Win32`); Aura-Call is not setting a custom `--user-agent`. But the live page does report `navigator.webdriver === true` under the current debug-attached Windows launch path. So if a service flags these managed debug profiles as bots, the stronger current suspect is the WebDriver/debug signal, not the UA string.
- Remaining auth caveat (2026-03-26): a fresh Windows-managed Chrome profile seeded from `C:\\Users\\ecoch\\AppData\\Local\\Google\\Chrome\\User Data\\Default` can now launch stably enough to complete a Grok browser run on this machine, but the resulting Grok tab still came up guest-only. Direct live proof on the managed Windows session: `BrowserAutomationClient.getUserIdentity()` returned `null`, and the Grok page still showed visible `Sign in` / `Sign up` CTAs. So “Chrome launched and answered a prompt” still does not prove first-run Grok auth from the imported Windows profile.
- Working workaround (2026-03-25): from WSL, Aura-Call can now attach to a Windows Chrome listening on Windows loopback with `--remote-chrome windows-loopback:<port>`. This uses a local WSL relay plus a Windows PowerShell helper over stdio, so it does not depend on raw WSL->Windows CDP TCP reachability.
- Historical direct-TCP notes (2026-03-25): earlier firewall and `portproxy` experiments are now fallback/debug material only. They are still relevant if you are manually exposing raw CDP from Windows Chrome to other machines, but they are no longer part of Aura-Call’s primary WSL -> Windows launch path.
- Direct-CDP caveat: if you intentionally bypass Aura-Call’s integrated path and expose a fixed Chrome port yourself, remember that mirrored networking/Tailscale can make Windows and WSL share the same apparent host IPs. In that mode, Windows-local success does not prove WSL can reach the same host:port, and a host-side `portproxy` on the same port Chrome wants can still block Chrome from binding CDP at all.
- WSL Chrome fallback (2026-03-18/19): when you explicitly choose WSL Chrome (`AURACALL_WSL_CHROME=wsl` or config `wslChromePreference: "wsl"`), Aura-Call now resolves `/usr/bin/google-chrome` + the WSL cookie DB even if Windows Chrome paths are still saved in config, keeps DevTools on local `127.0.0.1`, and bootstraps/reuses managed Aura-Call profiles under `~/.auracall/browser-profiles/...` instead of disposable `/tmp` profiles.
- Alternate-source bootstrap (2026-03-25): `--browser-bootstrap-cookie-path` / `browser.bootstrapCookiePath` lets WSL Chrome runs seed the managed Aura-Call profile from a different Chromium source such as Windows Brave. The managed bootstrap now copies only a selective auth-bearing Chromium subset instead of cloning the full browser profile, which keeps first-run setup around seconds instead of minutes for large Windows profiles.
- Windows Brave on WSL still has a hard limit on this machine: `/mnt/c/.../Brave-Browser/User Data/Default/Network/Cookies` throws `EACCES` for both direct file copy and Sweet Cookie reads. Aura-Call can still copy the rest of the Brave auth state into the managed profile, but Grok comes up guest-only because the actual cookie DB remains unreadable. Workaround: sign in once in the Aura-Call-managed profile, or export/provide cookies another way.
- WSL local Chrome launches must not reuse the Windows host IP. `launchChrome` now only uses `resolveWslHost()` for Windows-hosted Chrome; WSL Chrome should stay on local `127.0.0.1` CDP.
- Historical raw-CDP warning: do not bind a Windows `portproxy` on the same port Chrome itself must use for `--remote-debugging-port`. A host-side proxy like `192.168.50.108:45871 -> 127.0.0.1:45871` can prevent Chrome from opening `45871` at all. Use separate ports if you are doing that style of debugging outside Aura-Call.
- Chrome DevTools via mcporter: `chrome-devtools` server needs `CHROME_DEVTOOLS_URL` from a live session; without it `mcporter call chrome-devtools.*` fails. Expect this to be unset on Windows unless you bring your own Chrome session/URL.
- agent-scripts bash helpers: `runner`/`scripts/committer` can fail under PowerShell/CMD because of CRLF and bash expectations. If they explode, run commands directly (`pnpm ...`, `git add/commit`) instead.
- browser-tools binary: not built in `agent-scripts/bin` on Windows; `pnpm tsx scripts/browser-tools.ts` also fails there (no package manifest). Use a macOS-built binary or run from macOS if you need it.
- Prefer PowerShell + pnpm directly; watch for CRLF warnings when touching tracked files.

Future Windows gotchas belong here. Update this doc when you learn something new.
